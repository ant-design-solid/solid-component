`Polymorphic` is a very small utility component that delegates the final rendered element to the
caller through `as`.

## When to use

- You want a shared component API without hard-coding the final tag.
- You need to render as `button`, `a`, `div`, or a custom component based on context.
- You want the target element's attributes to stay visible at the type level.

## Rendering custom components

`as` can point to a custom component, not just a native tag. That makes the primitive useful for
wrapping router links, styled wrappers or app-level abstractions without losing a unified calling
shape.

```tsx
import { Polymorphic } from "@s-components/polymorphic";

function AppLink(props) {
  return <a {...props} data-kind="app-link" />;
}

export default function Example() {
  return (
    <Polymorphic as={AppLink} href="/motion">
      Open motion docs
    </Polymorphic>
  );
}
```

## Exported types

- `Polymorphic` performs the actual runtime rendering.
- `PolymorphicAttributes<T>` defines the `as` attribute.
- `PolymorphicProps<T, Props>` merges target-element props with your own component props.
- `ElementOf<T>` resolves the DOM element type behind a target component.

## Notes

- `as` is optional in the type layer but required by runtime behavior unless the caller sets a default.
- Prop passthrough does not add semantic fixes. Rendering as `a` does not magically make `disabled`
  meaningful.
- The package is intentionally small. Styling, accessibility adjustments and interaction policies
  belong in the higher-level component you build on top of it.

<Callout tone="note" title="Keep it low-level">
  If you find yourself adding behavior, variants and accessibility fixes directly into
  `Polymorphic`, you are likely moving responsibility into the wrong layer. Keep this primitive
  narrow and compose richer components on top.
</Callout>
